I.ARCHITECTURE · WHAT SAMARTH PLUS IS, AS A SYSTEM · Store Intelligence001 / 007
02 · Architecture · what Samarth Plus is, as a system

Node + Prisma + Postgres on Reliance infra. SCORM player, SSO bridge, signed-URL storage, single-session auth.

Samarth Plus is served inside the Reliance People First app. Entry is a notification on the People First home; tapping it opens the Samarth Plus interface inside the host app. Below is the slice of stack that runs the eligibility, learning, OJT and certification flows. Source · Module Work Breakdown · Raunak Sharma · Phase 1-6.

Lens 01 · Surface

People First entry · SSO bridge.

  • · Notification on People First home opens Samarth Plus
  • · validateJwt / createJwt exchange with PeopleFirst
  • · BU code + role propagated from PeopleFirst session
  • · HRMS sync gated to SSO users only
Lens 02 · Eligibility engine

BU master + role-skill matrix.

  • · Eligibility criteria per role · activated-certification + dynamic requirement lookup
  • · BU master table drives role mapping (replaces free-text business-name normalisation)
  • · Career tree reconciled across HRMS + user profile + journey config
  • · Role-skill matrix governs the TA → K1 → K2 → K3 → DM ladder
Lens 03 · Learning + OJT

SCORM player · signed-URL storage · journey-scoped state.

  • · SCORM 1.2 + 2004 manifest parsing · chunked upload (50-500MB) · CSP-safe iframe sandbox
  • · Signed-URL upload pipeline (storage.ts · storageService.ts) replaces Supabase Storage
  • · Progress wired to user_journey_progress · cmi.completion_status + success_status
  • · OJT activities scoped to active journey · prevents cross-pollution between enrolments
Lens 04 · Admin tower + auth

Bulk tooling · 2FA · audit trail.

  • · Excel + plain-text bulk upload for users · model stores · eligibility · OJT applicability
  • · Per-row error report · admins fix and re-upload only rejected rows
  • · Email-OTP 2FA (auth_2fa_challenges) · single-session via user_sessions · revoked-tokens table
  • · Audit trail logging · Fortify + BlackDuck cleared · CSP + same-site cookies
Stack at a glance

Frontend · React inside the People First webview. Backend · Node.js + Express + Prisma + TypeScript. Datastore · Postgres on Reliance infra, behind the Reliance gateway. Pipeline · Azure DevOps · Harbor registry · ADC credentials. Mail · Sendgrid. Source of truth for the Apr-2026 cutover · supabase-backup-20251014.tar.gz migrated into the new Postgres via a one-off backfill script.

II.WHAT'S LIVE · THE CANDIDATE JOURNEY · Store Intelligence002 / 007
03 · What's live · the candidate journey

Six steps from "I want my next role" to certified.

The journey an associate sees inside the People First app. Entry point: a notification on the People First home. "You are eligible to apply for a new role." Tapping it opens the Samarth Plus interface. The HRBP playbook below is what Reliance HRBPs use to brief frontline teams.

Step 01
Live

Pick Your Path.

Career Road Map shows the role lattice for the associate's format · TA / K1 / K2 / K3 / DM. Associate selects a target role, reviews responsibilities + skills gap, taps Begin the Program.

Constraint · within-format growth + cross-format exploration both allowed
Step 02
Live

Quick Check.

Pre-screening · Aptitude Test (MCQ, 30 min) + Behavioural Assessment (scenarios, 40 min). Proctored: front-camera identity capture. Quiet, well-lit room. 2 min per question, no go-back.

Constraint · screening must be cleared before the Learning Journey opens
Step 03
Live

Skill Up.

Learning Journey · SCORM-compliant skill modules auto-mapped to gaps for the target role. Mandatory: 100% module completion before progression. Skill assessments per module · re-attempts allowed.

Constraint · mandatory · 100% progress required to unlock OJT
Step 04
Live

Practice · OJT.

On-the-Job Training in a model store. Initiation → Shadowing → Checklist → Performance → Video Proof → Assessor Approval. Daily activity log; assessor reviews each day before approving.

Constraint · required for higher-leadership transitions (e.g. K3 → DM)
Step 05
Live

Get Certified.

All OJT days approved → status moves to Certified. Associate downloads a Certificate of Completion (PDF). Issuance verified on prod traffic during the Apr-2026 cutover.

Constraint · all sub-modules must be Approved
Step 06
Live · external handoff

Progress.

Associate brings the certificate to the Store Manager to discuss openings matching certified skills. HRBP coordinates the move. LMS + IJP integration on the forward roadmap will close this loop inside the platform.

Constraint · handoff to Store Manager + HRBP today; in-platform IJP on roadmap
III.PLATFORM PILLARS · WHAT'S WIRED IN · Store Intelligence003 / 007
04 · Platform pillars · what's wired in

Four employee pillars + an admin tower.

Pillar 01

Proctored Pre-Screening.

Aptitude + behavioural assessments with built-in proctoring, configurable timers per role, and prescreening-attempts persistence so an interrupted session resumes cleanly.

Pillar 02

Skill-Based Learning.

Auto-mapped SCORM-compliant courses targeting specific skill gaps per transition. Player handles SCORM 1.2 + 2004, chunked uploads (50–500MB), CSP-safe iframe, landscape orientation on tablets.

Pillar 03

Guided OJT.

Structured on-the-job training at model stores · proof-based activity checklists · day-by-day assessor review · video-proof upload · slot booking. OJT activities are scoped to the active journey · a user enrolled in two journeys sees only the current one's pending tasks.

Pillar 04

SSR (Assessor) Workflow.

Dedicated dashboard for buddies and trainers to validate proofs, approve days, and manage the assessor approval queue in real time.

Admin Control Tower

"God view" of the talent pipeline.

Role-Skill Matrix · transition logic across the TA → DM ladder · real-time view of who is Certified and ready · pipeline visibility per format and per region · bulk-upload tooling for users, model stores, eligibility, OJT applicability matrix · per-row error reporting so admins fix and re-upload only rejected rows.

IV.SHIPPED · 7-MONTH DELIVERY FROM SUPABASE PROTOTYPE TO REL... · Store Intelligence004 / 007
05 · Shipped · 7-month delivery from Supabase prototype to Reliance prod

Six phases. Dev → SIT → UAT → Prod. Oct-2025 to Apr-2026.

Incremental delivery log, by phase. Each phase names what was built, the environment it ran in, and one headline. Source · Module Work Breakdown · Raunak Sharma · primary contributor, with Vikas Singh and Yash on UI / SSO collaboration.

Phase 01 · Oct-2025
Dev only

Supabase to Node.js + Prisma migration.

  • · Stack before · Supabase (Postgres + Auth + Storage + Edge Functions)
  • · Stack after · Node.js + Express + Prisma + Postgres on Reliance infra
  • · Schema port · every Supabase table + RLS converted to Prisma models with application-level authorization
  • · Edge functions to REST routes · admin user CRUD, OJT seed, proof upload, prescreening calc, certificate issue, eligibility refresh
  • · Auth re-build from scratch · email-OTP, phone-OTP, JWT issuance, refresh handling
Phase 02 · Nov-2025
Dev only

SCORM, ILT, SSO, Reliance branding.

  • · SCORM chunked upload (50-500MB) with progress and resume
  • · Manifest parsing across SCORM 1.2 + 2004 · asset path rewriting · iframe sandboxing
  • · Progress tracking back to backend · cmi.completion_status, cmi.success_status
  • · ILT tables and instructor dashboard · disabled mid-month while OJT short-circuited
  • · SSO with PeopleFirst · validateJwt / createJwt exchange · BU code + role propagation
Phase 03 · Dec-2025 to Jan-2026
SIT

Security hardening · Fortify + BlackDuck loop.

  • · CSP headers + same-site cookies · audit-trail logging service · secure encryption for sensitive payloads
  • · Single-session enforcement via user_sessions · revoked-tokens table
  • · Server-side input validation · file-upload restrictions · max-login-attempt lockout
  • · All Fortify Critical + High items cleared · BlackDuck dependency upgrades · SSL cert refresh
  • · 2FA scaffolding added late Jan, reverted, re-implemented properly in Phase 4
Phase 04 · Feb-2026
UAT

2FA proper, journey-scoping, stabilization.

  • · Email-OTP 2FA after primary login · auth_2fa_challenges table · 60s expiry, resend, timer alignment
  • · Journey-scoping migration touched every read path · OJT, prescreening, learning all scoped to active journey
  • · OJT dashboard fixes · video completion progress · certification PDF render path
  • · HRMS sync for current BU code + tenure days · current_role_tenure_days migration
  • · Real Reliance HR + L&D reviewers exercising the build on UAT
Phase 05 · Mar-2026
UAT to pre-prod

Bulk-upload tooling for the prod data lift.

  • · Excel template upload for users · model stores · addresses · eligibility requirements
  • · Plain-text fallback when admins paste data instead of using the template
  • · Optimisation pass · 10k+ row chunked processing · per-row error report
  • · Assessment retake matrix · second-attempt logic · rebound time after failure
  • · Prescreening polish · attempts persistence · prescreening tests config
Phase 06 · Apr-2026
Prod · limited pilot

Prod cutover, mass data load, pilot users on real traffic.

  • · Model stores · entire Reliance store master, region- and BU-tagged
  • · Business Unit master table (20260428_add_business_unit_master) · finally fixes the business-name normalisation problem
  • · Skills + applicability matrix per BU per role · OJT applicability matrix (20260415_seed_ojt_grocery_fl)
  • · Eligibility checks updated for activated certifications + dynamic requirement lookup
  • · Sendgrid mail delivery · SSO end-to-end · certificate issuance verified on prod traffic
Cross-cutting challenges · what recurred across phases
  • · Populating prod data was the single biggest non-coding cost · formalised in Phase 6 but felt across every phase
  • · Career Tree had three independent sources of truth · HRMS, user profile, journey config · clean only after the Apr-2026 BU master
  • · SCORM packages from third-party vendors did not follow a single spec strictly · custom manifests, broken relative paths
  • · Migration parallel-run · Supabase kept alive while Node matured · every Oct feature shipped twice
  • · Fortify scan loop · long turnaround per cycle · the rate-limit was scan cycles, not coding speed
  • · Journey-context propagation made every read path journey-scoped · bugs leaked through to UAT for weeks
  • · Reliance infra constraints · Harbor registry, ADC credentials, restricted egress, SSL cert rotation · coordination with Reliance ops, not pure dev work
  • · Environment promotions had cost · every Dev → SIT → UAT → Prod move redid env-var plumbing, DB-URL handling, Sendgrid keys, and tester handoff
