1. PROJECT GOAL
================================================================================

AUTOMATE GARMENT MANUFACTURING COST ESTIMATION using AI-powered multi-agent
system that analyzes product images and specifications to generate detailed
Bill of Materials (BOM) with accurate pricing.

KEY OBJECTIVES:
- Speed: Generate cost estimates in 45-80 seconds vs 2-3 hours manual work
- Accuracy: 85%+ confidence using real market data (751K+ records)
- Coverage: Support 77+ garment categories across mens/womens/kids segments
- Intelligence: AI vision + semantic matching + cascading fallback strategies
- Reliability: Redis job queue eliminates 504 timeout errors
- Scalability: Async processing supports multiple concurrent requests

WHAT'S NEW IN V4:
✅ Temporal Workflow Engine — durable BOM pipeline with crash recovery + retries
✅ Techpack-First Costing — accept techpack PDFs as primary input (no images required)
✅ Smart Image Selection — Gemini ranks techpack images, filters reference/inspiration
✅ Dual-Flow Techpack — Flow A (text+images) and Flow B (text-only, skip vision)
✅ Service Cost Sanity Check — benchmark validation prevents anomalous DB prices
✅ Prod Deployment — env-aware GCS/CDN config (SIT + Prod from same codebase)
✅ Batch Processing — 274 techpacks processed, compared against source of truth

4.2 DATABASE AGENT (adk_database_agent.py + master_db.py)
──────────────────────────────────────────────────────────
Purpose: Intelligent querying with semantic matching + LLM validation

Key Features:
- Semantic Normalization: "Poplin" → "Polyester" (fabric name ≠ fiber)
- Fuzzy Matching: Find closest match when exact doesn't exist
- LLM-Constrained: Only returns values that exist in DB (no hallucination)
- Price Validation: Queries GPT-5.2 for market price ranges
- 24-hour Cache: Prevents redundant LLM API calls
- NULL GSM Handling: Cascade logic handles missing GSM values

LLM Price Validation Flow:
  1. Query DB for Poplin 100% Cotton prices
  2. Get median: ₹127/kg (from 6 records)
  3. Query GPT-5.2: "What is Poplin 100% Cotton market price in India?"
  4. GPT-5.2 responds: "₹400-500/kg"
  5. Validation: ₹127 is only 25% of ₹400 (WAY TOO LOW!)
  6. Decision: use_llm
  7. Final price: ₹450/kg (LLM mid-point)
  8. Cache result for 24 hours

4.3 CM CALCULATION ENGINE (V3 - Parameter-Based Dual Mode)
──────────────────────────────────────────────────────────
File: src/calculator/bom_generator.py

**NEW:** Two calculation modes via `use_efficiency_based_cm` parameter

MODE 1: SIMPLE (DEFAULT)
  Formula: CM = SMV × Rate
  Rates by Product Group:
    - KNIT: ₹3.2/minute
    - WOVEN: ₹3.5/minute
    - DENIMWEAR: ₹4.0/minute

  Example:
    SMV = 14 minutes, Product Group = KNIT
    CM = 14 × 3.2 = ₹44.8

  Use Case: Quick estimates, predictable pricing

MODE 2: EFFICIENCY-BASED
  Enabled via: use_efficiency_based_cm=true

  Process:
    1. Query CM database by segment/garment/SMV
    2. Apply factory tier multiplier:
       - tier1_metro: 1.2x
       - tier2_city: 1.0x
       - tier3_rural: 0.8x
    3. Calculate efficiency percentage
    4. SMV Verification (LLM-based):
       - Compare vision SMV with calculated SMV
       - Flag outliers >30% deviation
       - Request manual review if needed
    5. Apply outlier detection (>3× median → reject)
    6. Return final CM with confidence score

  Example:
    SMV (vision) = 14 minutes
    SMV (calculated) = 18 minutes
    Deviation = 28.6% (ACCEPTABLE, <30%)
    Factory Tier = tier1_metro
    Base CM = ₹42
    Final CM = ₹42 × 1.2 = ₹50.4

  Use Case: Accurate pricing with factory considerations

  **SMV Outlier Detection:**
    - Compares vision SMV vs calculated SMV
    - Threshold: 30% deviation
    - If exceeded: Adds warning to BOM
    - Uses GPT-5.1 (primary) / Gemini 3 Pro Preview (fallback)

4.6 TECHPACK-FIRST COSTING (V4)
────────────────────────────────
Files: src/temporal/techpack_activity.py, src/temporal/techpack_merger.py

Purpose: Accept techpack PDFs as primary input, no images required

Two Flows (user-controlled via use_techpack_images toggle):
  Flow A (Text + Images): Parse techpack text → extract images → Gemini ranks
    images against garment description → select best colourway → run vision
    on selected image → merge techpack specs with vision output → BOM
  Flow B (Text-Only): Parse techpack text → construct garment specs directly
    from extracted data → skip vision → pricing → BOM

Techpack Parsing Pipeline:
  CALL 1: PyMuPDF text extraction → Gemini 3.1 Pro structured parsing
    → garment type, fabric, GSM, composition, trims, services, colorways
  CALL 2: PyMuPDF image extraction → Gemini 3.1 Pro image ranking
    → Each image scored against garment name, colour, artwork description
    → Filter out reference/inspiration images of different garments
    → Priority: colourway_front + colourway_back > sketch > nothing

Spec Merge Logic (techpack_merger.py):
  - Techpack values OVERRIDE vision inferences (explicit > inferred)
  - Services from techpack BOM supplement vision-detected services
  - AOP/scatter prints detected from description keywords
  - Output format matches vision_analysis_activity output exactly

Tested: 274 techpack PDFs, 260 completed, compared against source of truth
  Single-piece accuracy: median +4%, 85% within ±50%

5. FABRIC PRICING CASCADE STRATEGY (V3 - 6 LEVELS + LLM VALIDATION)
================================================================================

Progressive Relaxation: From exact match to benchmark fallback
LLM validation on Levels 1-4, NULL GSM handling on Levels 3-4

LEVEL 1: EXACT MATCH (95% confidence)
──────────────────────────────────────
Filters: Fabric type + Composition + GSM (exact) + MOQ (exact)
LLM Validation: Applied if result found
Data Freshness: Last 12 months preferred

LEVEL 2a: RELAX MOQ ±100 (90% confidence)
──────────────────────────────────────────
Filters: Fabric + Composition + GSM (exact) + MOQ (±100)
LLM Validation: Applied if result found

LEVEL 2b: ANY MOQ (88% confidence)
───────────────────────────────────
Filters: Fabric + Composition + GSM (exact) + Any MOQ
LLM Validation: Applied if result found

LEVEL 3: RELAX GSM ±50 (85% confidence) **WITH NULL HANDLING**
───────────────────────────────────────────────────────────────
Filters: Fabric + Composition + GSM (±50 OR NULL) + Any MOQ
**ACCEPTS NULL GSM VALUES!**

SQL Example:
  WHERE gsm BETWEEN 150 AND 250 OR gsm IS NULL

Use Case:
- Some fabrics (e.g., Poplin) have NULL GSM in database
- Level 3 catches these cases
- LLM validation ensures price is reasonable

LEVEL 4: PRIMARY FIBER ONLY (75% confidence) **WITH NULL GSM**
───────────────────────────────────────────────────────────────
Filters: Fabric + Primary fiber (ignore blends) + NO GSM filter
Example: "60% Polyester 40% Cotton" → Search for "100% Polyester" only
LLM Validation: Applied if result found

LEVEL 5: BLENDING LOGIC (60% confidence)
─────────────────────────────────────────
Calculate weighted average from pure fiber prices
No GSM filter, fallback to all-time data if needed

LEVEL 6: BENCHMARK FALLBACK (50% confidence)
─────────────────────────────────────────────
Use universal specs historical consumption data
Last resort when no database matches found

================================================================================
6. LLM PRICE VALIDATION SYSTEM
================================================================================

Purpose: Validate database prices against real-world market estimates

Flow:
1. DATABASE QUERY (Any Level 1-4)
   - Retrieve price from fabric_master
   - Calculate median/average from multiple records

2. LLM QUERY (GPT-5.2)
   - Ask: "What is [fabric] [composition] [GSM] market price in India?"
   - Response format: "₹X-Y per kg" or specific estimate

3. VALIDATION LOGIC
   - Compare DB price with LLM price range
   - If DB price <50% of LLM minimum → use_llm
   - If DB price >200% of LLM maximum → use_llm
   - If within range → use_db
   - Apply adjustment factor if needed

4. CACHING (24-HOUR TTL)
   - Cache key: fabric + composition + gsm
   - Prevents redundant LLM API calls
   - Reduces cost and latency

12. KEY TECHNICAL DECISIONS (V3)
================================================================================

1. PARAMETER-BASED CM CALCULATION
   - Allows users to choose speed (simple) vs accuracy (efficiency)
   - Simple mode: Predictable, fast, no database queries
   - Efficiency mode: Considers factory tier, includes SMV verification
   - Default: simple (backward compatible)

2. PRODUCT GROUP HINT PRIORITY
   - User hint > Vision classification > Specs fallback
   - Critical for woven vs knit distinction
   - Affects fabric type suggestions and CM rates

3. SIMPLIFIED THREAD FORMULA
   - Old: Complex meter-based calculation with GCS lookup
   - New: Direct SMV/5 formula
   - Reason: Industry standard, reduces complexity

4. SMV OUTLIER DETECTION
   - Compares vision SMV vs calculated SMV
   - 30% threshold chosen after analyzing historical data
   - Prevents major cost estimation errors

5. DEBUG INFO BACKWARD COMPATIBILITY
   - Handles old data (session state as string)
   - Handles new data (session state as JSONB)
   - Safe fallback to empty {} prevents API errors

6. DESIGN GARMENT ORDERING
   - ORDER BY created_at DESC ensures latest first
   - Critical for user experience (recent work appears first)
